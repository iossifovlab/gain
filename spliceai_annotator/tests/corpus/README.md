# SpliceAI differential-test corpus

A frozen hg38 / GENCODE fixture corpus that pins **all 16** `spliceai_annotator`
attributes from today's TensorFlow output, so the upcoming ONNX migration
(iossifovlab/gain#297, #299) has real ground truth to be checked against.
See iossifovlab/gain#320.

## Layout

| Path | What it is | Regenerated by |
|------|------------|----------------|
| `build_corpus.py` | Cuts + rebases sequence windows and gene-model rows from the real hg38/GENCODE GRR into the fixture GRR, and writes the variant manifest. | run manually (needs the real GRR) |
| `corpus_manifest.json` | Every corpus variant: local contig, position, ref/alt, intended `distance`, category, expected disposition. Also per-contig metadata (`length`, `real_chrom`, `window_start`) so a variant can be de-rebased to real hg38 coordinates ‚Äî `real_pos = local_pos + window_start - 1` (used by the #321 integration tier; inert here). | `build_corpus.py` |
| `../fixtures/hg38/genome` | `genome` resource ‚Äî the cut+rebased local contigs. | `build_corpus.py` |
| `../fixtures/hg38/gene_models` | `gene_models` resource ‚Äî rebased real transcripts + two crafted transcripts (mixed-strand + near-end axes). | `build_corpus.py` |
| `regenerate_baseline.py` | Annotates every variant at its distance through `pipeline.annotate` (all 16 attributes) and writes the committed baseline. | run manually (needs the **TensorFlow** venv) |
| `baseline.json.gz` | The TensorFlow ground truth: `{variant_id: {attribute: value}}`. gzipped (the distance=5000 probability vectors are large). | `regenerate_baseline.py` |
| `pipeline_config.py` | Single source of truth for the pipeline YAML + tiering constants, shared by the baseline generator and the differential test. | ‚Äî |
| `rebasing.py`, `fixture_io.py` | Pure coordinate rebasing + fixture emission helpers (unit-tested in `../test_corpus_rebasing.py`). | ‚Äî |

The differential test that consumes all of this is
`../test_spliceai_differential.py`. It is marked `@pytest.mark.integration`
(~10 min under TensorFlow), so it runs in the **gain-spliceai-integration**
downstream job, not the fast per-PR `spliceai_annotator` step (which runs
`-m "not integration"`). See `spliceai_annotator/pytest.ini`,
`Jenkinsfile.integration`, and the `Trigger spliceai integration` stage in the
root `Jenkinsfile`.

## Why the baseline is generated *only* from TensorFlow

The baseline is the ground truth the ONNX migration is validated against.
iossifovlab/gain#299 **removes TensorFlow** from the runtime. After that, the
baseline can be regenerated only through this documented TF path
(`regenerate_baseline.py` under the TF venv). Regenerating it from ONNX would
silently redefine "correct" as "whatever ONNX now does" and discard the ground
truth this harness exists to protect. The `.h5` model files stay as source of
truth for both.

## Why the fixture rebases every locus to a local contig

`build_corpus.py` windows each locus and rebases the gene-model rows **and** the
variant positions onto a local contig that starts at 1. Rebasing shifts every
coordinate by the same offset, so a variant's position *relative to* the
transcripts around it ‚Äî the only thing the annotator's padding math depends on ‚Äî
is preserved, while a deep locus (e.g. `chr21:46M`) does not blow up into a
chromosome-sized N-padded FASTA. Every locus is flanked by ‚â• `width//2` at
distance=5000 (10.1 kb), so any in-gene position is a valid, full-window
annotatable at every distance.

## Regenerating

```bash
# from the spliceai_annotator dir, inside the worktree TF venv
# 1. re-cut the fixture + manifest from the real GRR (GRR_ROOT overridable)
python -m tests.corpus.build_corpus
# 2. re-pin the TensorFlow baseline
python -m tests.corpus.regenerate_baseline
```

## Corpus axes

Both strands; multi-gene loci (`SON`+`DONSON` overlap) and multi-transcript
genes (`SON`, `DONSON`, `COL6A2`); a crafted mixed-strand gene; `distance ‚àà
{0, 50, 500, 5000}`; SNV + short deletion + a boundary deletion
(`del_len == distance`, the largest one still annotated) + short/long
insertion; every `_is_valid_annotatable` rejection (complex, ref-too-long,
alt-too-long, strange alt, wrong ref, near-chromosome-end, **and
deletion-longer-than-distance**); null variants **and** splice-site variants.
At `distance=0` every variant is rejected (`ref` longer than `2*distance == 0`)
‚Äî pinned as all-None.

## Why long deletions are refused (not annotated)

A deletion whose removed span exceeds the model half-window
(`ref_len - 1 > distance`) is **refused** by the annotator (all-None), by
design. Beyond that point `_prediction_padding_batch` mis-reconstructs the alt
window: the sequential path reproduces the original Illumina SpliceAI exactly,
but the batch path does not (e.g. a 90 bp deletion at `distance=50` gave
`DP_AG` batch `-17` vs sequential/Illumina `-43`). Rather than emit values that
depend on which code path ran, `_is_valid_annotatable` rejects these records ‚Äî
so `batch_annotate == annotate` then holds **exactly** for every annotated
variant, and the `reject_del_too_long` / `del_boundary` axes pin both sides of
that threshold.
