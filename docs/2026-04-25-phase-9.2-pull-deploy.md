# Plan: Phase 9 (slice 2) — pull-deploy on prod hosts

## Context

Phase 9.1
(`docs/2026-04-25-phase-9-image-push.md`) wired the root
`Jenkinsfile` to push the wheel-based backend image and the
Apache-based frontend image to `registry.seqpipe.org` on
every master build, with three tags per repo: `:${BUILD_NUMBER}`,
`:${GIT_SHORT}` (8-char SHA), and `:latest`.

This slice flips the prod-host compose files
(`web_infra/compose-iossifovweb.yaml` and
`compose-wigclust.yaml`) so they pull those images from the
registry instead of expecting a local-build tag.

Before 9.2 the operator workflow was effectively:

```bash
# On the prod host: rebuild images from a local checkout
git pull
docker build -f web_api/Dockerfile.production \
    -t gain-web-api-prod:latest .
docker build -f web_ui/Dockerfile.production \
    --build-arg BACKEND_IMAGE=gain-web-api-prod:latest \
    -t gain-web-ui-prod:latest .
docker compose -f web_infra/compose-iossifovweb.yaml up -d
```

After 9.2:

```bash
# On the prod host: just pull the images Jenkins already
# built and pushed.
docker compose -f web_infra/compose-iossifovweb.yaml pull
docker compose -f web_infra/compose-iossifovweb.yaml up -d
```

## Scope

**In scope:**

- Change the `BACKEND_IMAGE` default fallback in both prod
  compose files from `gain-web-api-prod:latest` (local-build
  tag) to `registry.seqpipe.org/gain-web-api:latest`.
- Change the `FRONTEND_IMAGE` default fallback similarly
  from `gain-web-ui-prod:latest` to
  `registry.seqpipe.org/gain-web-ui:latest`.
- Keep the `${BACKEND_IMAGE:-...}` / `${FRONTEND_IMAGE:-...}`
  env-var indirection so an operator can pin to a specific
  tag (e.g.
  `BACKEND_IMAGE=registry.seqpipe.org/gain-web-api:abcd1234
  docker compose -f compose-iossifovweb.yaml pull`) for a
  controlled rollout or a quick rollback to a known-good
  build.

**Out of scope:**

- **`compose-jenkins.yaml`** (e2e fixture stack used by the
  `gain-web-e2e` Jenkins job and by `web_infra/compose.yaml`
  for local dev). It deliberately defaults to a local-build
  tag so the e2e job tests exactly what it just built; if it
  pulled the registry's `:latest` instead, it could mask
  regressions in the current branch. See Phase 9.3 for a
  separate, more targeted change to make e2e pull a specific
  upstream-build tag.
- **`compose.yaml`** (local dev). Inherits from
  `compose-jenkins.yaml`, so its image defaults track
  whichever the developer just built.
- **A `pull-deploy` runbook** for operators. The two-line
  workflow above is short enough to live in this doc; if a
  longer ops doc emerges (rollback procedure, registry
  outage handling, etc.), it can land separately.
- **Pinning policy**. Whether prod should track `:latest`
  or always pin to a specific tag is an ops/policy
  decision; the env-var indirection supports either, and
  this commit picks the one that needs the smallest
  operator action (`pull && up -d` keeps tracking master).

## Approach

The change is mechanical: six string replacements across
two files. Both prod composes already use `image:` (no
`build:` directives to retire — that part was speculative
in the Phase 9.1 doc; turns out the conda-era composes had
`build:` directives but the Phase 8 rewrite replaced them
with `image: ${BACKEND_IMAGE:-...}` already).

`compose-iossifovweb.yaml`:

```diff
   backend-migrate:
-    image: ${BACKEND_IMAGE:-gain-web-api-prod:latest}
+    image: ${BACKEND_IMAGE:-registry.seqpipe.org/gain-web-api:latest}
   ...
   backend:
-    image: ${BACKEND_IMAGE:-gain-web-api-prod:latest}
+    image: ${BACKEND_IMAGE:-registry.seqpipe.org/gain-web-api:latest}
   ...
   frontend:
-    image: ${FRONTEND_IMAGE:-gain-web-ui-prod:latest}
+    image: ${FRONTEND_IMAGE:-registry.seqpipe.org/gain-web-ui:latest}
```

`compose-wigclust.yaml` is the same shape, three
substitutions in the same services.

## Verification

Local: run `docker compose -f web_infra/compose-iossifovweb.yaml
config` and confirm the resolved image strings point at
`registry.seqpipe.org/gain-web-{api,ui}:latest`.

Pre-deploy on iossifovweb (after 9.1's first master-build
push lands):

```bash
docker compose -f web_infra/compose-iossifovweb.yaml pull
docker compose -f web_infra/compose-iossifovweb.yaml \
    config --images
```

Both `pull` and `config --images` should report the registry
URLs and successfully fetch.

Production cutover on each host:

```bash
cd /path/to/gain    # whatever the deploy checkout is
git pull            # picks up this commit
docker compose -f web_infra/compose-iossifovweb.yaml pull
docker compose -f web_infra/compose-iossifovweb.yaml up -d
```

`up -d` is a no-op for any service whose image digest hasn't
changed; it recreates only the backend / backend-migrate /
frontend if the registry served a newer image.

To pin a specific build (rollback or controlled rollout):

```bash
export BACKEND_IMAGE=registry.seqpipe.org/gain-web-api:42
export FRONTEND_IMAGE=registry.seqpipe.org/gain-web-ui:42
docker compose -f web_infra/compose-iossifovweb.yaml pull
docker compose -f web_infra/compose-iossifovweb.yaml up -d
```

## Risks and known unknowns

- **Registry availability becomes a deploy-time dependency.**
  Before 9.2 a prod host could rebuild from a local git
  checkout if `registry.seqpipe.org` was down. After 9.2 the
  default flow needs the registry. Mitigation: any image
  already cached on the host stays runnable; only `pull`
  needs the registry. For an extended outage, operators can
  fall back to building locally (`BACKEND_IMAGE=$(docker
  build -q -f web_api/Dockerfile.production .)
  docker compose ... up -d`).
- **`:latest` is a moving pointer.** If two prod hosts pull
  at different times during a deploy, they may transiently
  run different images. Mitigation: in practice deploys are
  per-host and infrequent; for stricter guarantees, pin to
  a specific tag during the window.
- **No automatic prod rollout.** This slice only swaps the
  defaults; it doesn't add a webhook / CD pipeline that
  pulls automatically when a new `:latest` is pushed.
  Operator still has to ssh in and run `pull && up -d`. A
  CD slice is out of scope here.
