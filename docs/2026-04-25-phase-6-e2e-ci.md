# Plan: Phase 6 — Move web_e2e (Playwright) onto the root Jenkinsfile and retire web_infra/Jenkinsfile

## Context

Phases 1–5 of the gpf-web-annotation merge are done; master is
green. The root `Jenkinsfile` now owns CI for `core`,
`demo_annotator`, `vep_annotator`, `spliceai_annotator`,
`web_api`, and `web_ui` in one parallel block, plus a sequential
`Conda packages` stage that builds `gain-core`,
`gain-demo-annotator`, `gain-vep-annotator`, and
`gain-spliceai-annotator` conda artefacts into `dist/conda/`.

The remaining piece is `web_e2e` — a Playwright suite (5
`.spec.ts` files; ~1200s global budget; serial in CI) that
exercises the whole stack end-to-end via real Chromium against a
production `frontend` container talking to a production
`backend` container, MailHog, postgres, and a `gpf-image` that
holds the gain Python packages.

Today this flow lives in `web_infra/Jenkinsfile`. Its only
external coupling is an `upstream(...)` trigger on
`iossifovlab/gpf-conda-packaging/master` and a `copyArtifacts`
step that pulls a `conda-channel.tar.gz` from there into
`web_infra/conda-channel/` so the local `Dockerfile.gpf` can
install gain-core. **That coupling is going away in Phase 6**:
the e2e stage will be re-pointed at the conda channel that the
root build itself produces in `dist/conda/`. The e2e flow
therefore depends only on subprojects from this monorepo —
`core` (provides the `gain-core` conda package), `web_api`
(provides `Dockerfile.production` for `backend-e2e` and the
`django-gpf-web-annotation` wheel), and `web_ui` (provides
`Dockerfile.production` for `frontend-e2e`). No external job
gating, no copyArtifacts, no upstream trigger.

Today's `web_infra/Jenkinsfile` shape:

- `triggers { upstream(upstreamProjects: 'iossifovlab/gpf-conda-packaging/master', ...) }` — **dropped in Phase 6**.
- `Copy artifacts` — pulls `results/conda-channel.tar.gz` from
  the upstream job and untars into `web_infra/conda-channel/`. **Dropped in Phase 6** (replaced by promoting the root build's own `dist/conda/` artefacts into `web_infra/conda-channel/`).
- `Build` — `docker compose build ubuntu-image gpf-image`.
- `Build Backend image` — `docker compose build backend-e2e`
  (production `web_api/Dockerfile.production`).
- `Build Frontend image` — `docker compose build frontend-e2e`
  (production `web_ui/Dockerfile.production`).
- `Run E2E Tests` — `docker compose run --rm e2e-tests`, copies
  `web_e2e/reports/junit-report.xml` to
  `web_infra/reports/e2e-junit-report.xml`.

After Phase 5, every other stage in `web_infra/Jenkinsfile` is
gone, so once the e2e flow moves, the file becomes redundant.
Phase 6 moves it and `git rm`s `web_infra/Jenkinsfile` in the
same change.

## Scope

**In scope:**

- **Generate `web_e2e/package-lock.json`** inside the same
  Playwright base image the Dockerfile uses
  (`mcr.microsoft.com/playwright:v1.51.1-jammy`), so transitive
  resolution in the lock matches what CI installs. Stop ignoring
  the lock if `web_e2e/.gitignore` ignores it (mirrors the
  Phase 5 fix for `web_ui/.gitignore`).
- **Add `web_e2e/.dockerignore`** (none today). Mirror
  `web_ui/.dockerignore`: exclude `node_modules`, `dist`,
  `test-results`, `playwright-report`, `reports`, IDE files,
  `.git`, etc.
- **Switch `web_e2e/Dockerfile.playwright` to `npm ci`** for
  determinism (currently `npm install`). Keep the filename —
  there's no production variant of the Playwright image, so the
  `<project>/Dockerfile = CI image` convention doesn't apply
  here; `.playwright` is descriptive and worth keeping.
- **Add a `web_e2e` sequential stage to the root `Jenkinsfile`**,
  placed *after* the existing `Conda packages` stage and *before*
  `Trigger VEP integration`. The e2e flow needs the
  `gain-*.conda` artefacts that `Conda packages` writes to
  `dist/conda/` (those are the conda channel `gpf-image`
  installs from), so the stage cannot live in the same parallel
  block as `core` / `web_api` / `web_ui` — it depends on their
  build outputs being on disk. Sequential placement also keeps
  us from running three concurrent compose stacks (core's
  apache+minio, web_api's mail, web_e2e's full stack) on the
  same Jenkins agent.
  - Stage uses
    `COMPOSE_PROJECT="gain-ci-web-e2e-${env.BUILD_NUMBER}"` and
    a matching `COMPOSE_NETWORK` env so cleanup is targeted.
  - Inside a `try { ... } finally { ... }`:
    - **Promote `dist/conda/*.conda` to `web_infra/conda-channel/`**
      so `Dockerfile.gpf` finds the gain conda packages where
      it already expects them
      (`web_infra/conda-channel/<arch>/`). One line of `cp` /
      `rsync`; keeps `Dockerfile.gpf` unchanged.
    - `docker compose -p "$COMPOSE_PROJECT" -f web_infra/compose-jenkins.yaml build ubuntu-image gpf-image backend-e2e frontend-e2e e2e-tests`
    - `docker compose -p "$COMPOSE_PROJECT" -f web_infra/compose-jenkins.yaml up -d --wait backend-e2e frontend-e2e mail`
    - `docker compose -p "$COMPOSE_PROJECT" -f web_infra/compose-jenkins.yaml run --rm e2e-tests`
    - `mkdir -p reports/web_e2e && cp web_e2e/reports/junit-report.xml reports/web_e2e/junit.xml`
    - `finally`: `docker compose -p "$COMPOSE_PROJECT" -f web_infra/compose-jenkins.yaml down -v --remove-orphans || true`
  - `post { always { script { publishReports('web_e2e') } } }`
    publishes the JUnit. Also `archiveArtifacts` for
    `web_e2e/reports/**` so Playwright videos and traces are
    retained on failure.
- **No upstream trigger, no `copyArtifacts` from upstream**.
  The root pipeline's existing
  `options { timeout; buildDiscarder }` block stays untouched.
  E2E uses the in-repo `dist/conda/*.conda` artefacts as its
  conda channel — the same packages the root build already
  produces every run in the existing `Conda packages` stage.
- **Retire `web_infra/Jenkinsfile`**. With the e2e flow moved,
  every stage in `web_infra/Jenkinsfile` is now redundant
  (Start/Copy artifacts/Build/Build Backend image/Build
  Frontend image/Run E2E Tests/post all existed only to
  support e2e; the upstream trigger and copyArtifacts go away
  entirely rather than moving). `git rm` it.
- **Verify locally** that the new root `Jenkinsfile` runs the
  e2e suite end-to-end and produces a well-formed JUnit
  report.

**Out of scope:**

- Retiring conda env files / `web_api/Dockerfile.dev` /
  `backend-dev` (Phase 7).
- Retiring the upstream `gpf-conda-packaging` coupling
  (Phase 8 — optional).
- Disabling the legacy Jenkins job that pointed at
  `web_infra/Jenkinsfile` — Jenkins admin task; mention in the
  commit body but no code change in this repo.
- `compose.yaml` / `compose-iossifovweb.yaml` /
  `compose-wigclust.yaml` — production deploy compose files,
  unaffected.

### Root `Jenkinsfile` — sequential web_e2e stage after Conda packages

#### Pipeline options

Untouched. The existing block stays exactly as it is today:

```groovy
options {
    timeout(time: 1, unit: 'HOURS')
    buildDiscarder(logRotator(numToKeepStr: '20'))
}
```

No `triggers { upstream(...) }` is added — Phase 6 cuts the
external coupling to `iossifovlab/gpf-conda-packaging/master`.

#### Conda channel — *no* new stage

The existing `Conda packages` sequential stage already produces
`dist/conda/*.conda` (gain-core, gain-demo-annotator,
gain-vep-annotator, gain-spliceai-annotator) on every run. The
new `web_e2e` stage promotes these into the path
`Dockerfile.gpf` expects (`web_infra/conda-channel/`) at the
top of its script — no separate stage, just a one-line copy.

#### New `web_e2e` sequential stage

Placed between the existing `Conda packages` and
`Trigger VEP integration` stages:

```groovy
stage('web_e2e') {
    environment {
        COMPOSE_PROJECT =
            "gain-ci-web-e2e-${env.BUILD_NUMBER}"
        COMPOSE_NETWORK =
            "gain-ci-web-e2e-${env.BUILD_NUMBER}_default"
    }
    steps {
        script {
            try {
                sh '''
                    mkdir -p web_e2e/reports reports/web_e2e \\
                        web_infra/conda-channel/noarch
                    # Promote the in-repo conda artefacts into
                    # the path Dockerfile.gpf expects. Phase 6
                    # replaces the old upstream conda-channel
                    # tarball with the root build's own output.
                    cp dist/conda/*.conda \\
                        web_infra/conda-channel/noarch/
                    docker compose \\
                        -p "$COMPOSE_PROJECT" \\
                        -f web_infra/compose-jenkins.yaml \\
                        build ubuntu-image gpf-image \\
                              backend-e2e frontend-e2e \\
                              e2e-tests
                    docker compose \\
                        -p "$COMPOSE_PROJECT" \\
                        -f web_infra/compose-jenkins.yaml \\
                        up -d --wait \\
                            backend-e2e frontend-e2e mail
                    docker compose \\
                        -p "$COMPOSE_PROJECT" \\
                        -f web_infra/compose-jenkins.yaml \\
                        run --rm e2e-tests
                    cp web_e2e/reports/junit-report.xml \\
                        reports/web_e2e/junit.xml
                '''
            } finally {
                sh '''
                    docker compose \\
                        -p "$COMPOSE_PROJECT" \\
                        -f web_infra/compose-jenkins.yaml \\
                        down -v --remove-orphans || true
                '''
            }
        }
    }
    post {
        always {
            script {
                publishReports('web_e2e')
                archiveArtifacts(
                    artifacts: 'web_e2e/reports/**',
                    allowEmptyArchive: true,
                    fingerprint: false,
                )
            }
        }
    }
}
```

Notes:

- `compose-jenkins.yaml`'s `e2e-tests` service already has
  `command: ["npx", "playwright", "test"]` and `CI: "1"` env,
  and depends on `backend-e2e`/`frontend-e2e` healthchecks, so
  `compose run --rm e2e-tests` is the entire test invocation.
- `up -d --wait backend-e2e frontend-e2e mail` lets the
  healthchecks gate readiness before we start the e2e
  container; `compose-jenkins.yaml`'s healthchecks for
  `backend-e2e` (`curl http://localhost:9001/api/user_info`)
  and `frontend-e2e` (`curl http://localhost/`) are already
  defined.
- The `e2e-tests` service builds an image tagged
  `${COMPOSE_PROJECT}-e2e-tests`; `compose down -v` cleans it
  along with the volumes (`-v` removes the postgres
  `pg-data`/`static-data` volumes so they can't leak between
  builds). No new entry needed in the post-block cleanup
  loop.

#### Cleanup-loop note

`gain-web-e2e-ci` is *not* added to the
`for img in gain-core-ci gain-demo-annotator-ci ...` loop in
`post.cleanup` — the e2e stage doesn't tag a `gain-web-e2e-ci`
image (it uses compose with the `e2e-tests` service name). The
`compose down -v --remove-orphans` in the stage's `finally`
already removes everything compose owns.

### Retire `web_infra/Jenkinsfile`

is gone. `git rm web_infra/Jenkinsfile`. The associated Jenkins

(`Dockerfile.gpf`, `Dockerfile.ubuntu`, `Makefile`,
`README.md`) all stay — they still describe the production
deployment story and the e2e fixture stack.

## Risks and known unknowns

- **`/mnt/cephfs/seqpipe/grr` host mount**.
  `compose-jenkins.yaml`'s `backend-e2e` mounts
  `/mnt/cephfs/seqpipe/grr:/grr` from the host. The Jenkins
  runner that runs `web_infra/Jenkinsfile` today already has
  this mount (otherwise e2e wouldn't pass on master). The root
  `Jenkinsfile` runs on the same `agent any` so the mount is
  reachable, but on the first dry run we should verify (the
  failure mode would be a clear "no such file" from
  Django at startup). If it ever becomes a problem, swap the
  mount for a synthetic GRR fixture inside the e2e flow — out
  of Phase 6 scope unless we hit it.
- **`gpf-image` build time**. Several minutes from cold; this
  becomes the longest pole of the sequential `web_e2e` stage,
  on top of the e2e suite's own ~few-minute runtime.
  Acceptable — `gpf-image` benefits from docker layer cache
  across builds, and the sequential placement after
  `Conda packages` means the parallel block has already
  finished by the time e2e starts.
- **Sequential vs parallel placement**. Phase 6 puts `web_e2e`
  *after* the parallel `Sub-projects` and `Conda packages`
  stages because it depends on `dist/conda/*.conda` being on
  disk. The trade-off is wall-clock: e2e tacks onto the end
  rather than overlapping with sub-project CI. If wall-clock
  becomes a problem we can later pull the conda build for
  *just* the gain packages e2e needs into a parallel pre-step,
  but that's an optimisation, not a correctness concern.
- **Conda channel layout under `web_infra/conda-channel/`**.
  `Dockerfile.gpf` expects a conda channel layout (typically
  `noarch/` and `<arch>/` subdirs with `repodata.json`). The
  in-repo `dist/conda/*.conda` artefacts may need a
  `conda index` (or `rattler-build` equivalent) run after the
  copy so the channel resolves. Check `Dockerfile.gpf` during
  implementation; if a `rattler-build index` is needed, run it
  inside the existing `gain-conda-builder-ci` image (already
  built in the root pipeline) rather than introducing a new
  tool dependency.
- **E2E flake**. Browser tests are inherently flakier than
  unit tests; the existing 5 specs pass on master today, but
  if the move surfaces a new environmental flake the JUnit
  publisher will mark UNSTABLE. Same handling model as
  ruff/mypy hits: read the report, fix or stabilise.
- **`compose-jenkins.yaml` relative-context paths**. Services
  build from `context: ../web_api` etc., relative to the
  compose file's location (`web_infra/`). Invoking with
  `-f web_infra/compose-jenkins.yaml` from the repo root keeps
  the relative paths resolving correctly to the repo's
  `web_api/` / `web_ui/` / `web_e2e/`. Verified by reading the
  YAML; sanity-check on the first dry run.
