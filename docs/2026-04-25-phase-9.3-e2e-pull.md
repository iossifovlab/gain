# Plan: Phase 9 (slice 3) — gain-web-e2e pulls pre-built images

## Context

Phase 9.1 (`docs/2026-04-25-phase-9-image-push.md`) wired
the root `Jenkinsfile` to push the wheel-based backend image
and the Apache-based frontend image to
`registry.seqpipe.org` on every master build, tagging each
with `:${BUILD_NUMBER}` + `:${GIT_SHORT}` + `:latest`.

Before 9.3, the `gain-web-e2e` downstream job rebuilt those
exact images itself: copy the parent's wheel artefacts via
`copyArtifacts`, then `docker build -f
web_api/Dockerfile.production` and `docker build -f
web_ui/Dockerfile.production`. ~5 min per master e2e run
duplicating work the parent had just done.

This slice teaches `gain-web-e2e` to pull the parent's
already-pushed images on master triggers and retag them
locally, replacing the build-from-wheels work. Branch
triggers and manual runs still go through the existing
build path (the parent doesn't push for branches).

## Scope

**In scope:**

- Refactor `web_e2e/Jenkinsfile.e2e` to dispatch between
  two image-preparation paths in a single `Prepare
  backend/frontend images` stage:
  - **Pull path** (master triggered downstream of the
    parent — `BRANCH_NAME == 'master'` and
    `UPSTREAM_BUILD` non-empty): `docker pull
    registry.seqpipe.org/gain-web-{api,ui}:${UPSTREAM_BUILD}`,
    then `docker tag` to the local
    `$BACKEND_IMAGE` / `$FRONTEND_IMAGE` so the rest of the
    pipeline doesn't care which path produced the images.
  - **Build path** (branch triggered downstream OR manual
    run with no upstream link): the existing flow —
    `copyArtifacts` of the parent's wheels (or container-
    build them from source if no upstream link), then
    `docker build` locally.
- Pull out the e2e-tests Playwright image build into its
  own `Build e2e image` stage. Always built locally; not
  registry-pushed. This keeps the stage view in Jenkins
  showing pull/build vs Playwright build timings
  separately.
- Update the file-header comment to describe the two
  paths.

**Out of scope:**

- **Pushing branch images.** Phase 9.1 deliberately
  pushed only on master; Phase 9.3 inherits that. If a
  later slice pushes per-branch tags (e.g. `:branch-X-42`),
  the `canPull` gate here would relax to drop the
  `BRANCH_NAME == 'master'` check.
- **Pulling on manual runs without an upstream link.** A
  manual run could in principle pull `:latest` from the
  registry, but `:latest` doesn't necessarily match the
  COMMIT_SHA the run is testing. Safer to fall through to
  the build-from-wheels path; if a user wants to test the
  current `:latest`, they can do it outside the e2e job.
- **Aggressive cleanup of the pulled registry image.** The
  `post.cleanup` block still only `docker rmi`'s the local
  `$BACKEND_IMAGE` / `$FRONTEND_IMAGE` tags; the
  registry-prefixed image stays in the agent's docker
  cache so subsequent pulls hit the cache.
- **Auth wiring for the pull.** Build #45 demonstrated the
  agent already has `docker login registry.seqpipe.org`
  configured (the push worked without explicit
  credentials in the pipeline). If pull surfaces a 401 in
  practice, mirror Phase 9.1's `credentials()` env-block
  binding.

## Approach

The `Prepare backend/frontend images` stage is a single
script block branching on `canPull`:

```groovy
boolean canPull = params.BRANCH_NAME == 'master' \
    && params.UPSTREAM_BUILD?.trim()
if (canPull) {
    String tag = params.UPSTREAM_BUILD
    sh """
        docker pull 'registry.seqpipe.org/gain-web-api:${tag}'
        docker pull 'registry.seqpipe.org/gain-web-ui:${tag}'
        docker tag 'registry.seqpipe.org/gain-web-api:${tag}' \
            "\$BACKEND_IMAGE"
        docker tag 'registry.seqpipe.org/gain-web-ui:${tag}' \
            "\$FRONTEND_IMAGE"
    """
} else {
    // copyArtifacts (downstream) OR container-build wheels
    // (manual no-upstream); then docker build the prod images
    // locally with the $BACKEND_IMAGE / $FRONTEND_IMAGE tags.
}
```

`$BACKEND_IMAGE` and `$FRONTEND_IMAGE` (set in the
pipeline-level `environment` block) keep their existing
form: `gain-web-api-prod:${BUILD_NUMBER}` /
`gain-web-ui-prod:${BUILD_NUMBER}`. Compose-jenkins.yaml
reads these and the `post.cleanup` block `docker rmi`'s
them, neither needs to change.

## Verification

The pull path can't be exercised locally without network
access to `registry.seqpipe.org` and a tag the registry
serves. So:

**Local (build path only):**

Trigger a manual `gain-web-e2e` run with default
parameters (BRANCH_NAME=master, UPSTREAM_BUILD empty). The
canPull gate is false (UPSTREAM_BUILD empty), the build
path runs (container-builds wheels from source), the
Playwright suite runs against those locally-built images.
Same behaviour as before this commit.

**Jenkins-side:**

1. First master parent build after this lands: parent
   pushes images to registry (Phase 9.1, unchanged), then
   triggers `gain-web-e2e` with `BRANCH_NAME=master` and
   `UPSTREAM_BUILD=<parent build number>`. The e2e job's
   `Prepare backend/frontend images` stage logs `docker
   pull registry.seqpipe.org/gain-web-api:<n>` and
   succeeds in ~30s instead of ~5 min.
2. First non-master parent build after this lands:
   parent does NOT push (branch gating in 9.1), triggers
   `gain-web-e2e` with `BRANCH_NAME=<branch>`. The e2e
   job's canPull gate is false, the wheels+build path
   runs as before.
3. Total e2e wall-clock time on master should drop from
   ~10 min to ~5 min.

## Risks and known unknowns

- **Pull failure mid-trigger-chain.** If the pull fails
  (network glitch, registry down, tag missing), the e2e
  build fails loudly. Mitigation: re-run; if persistent,
  manually trigger gain-web-e2e with `UPSTREAM_BUILD=''`
  to fall through to the wheels+build path.
- **Cache pressure on the agent.** Pulled images
  accumulate in the docker cache across builds. Three
  registry-prefixed tags per master build × ~hundreds of
  builds → potentially many GB. Mitigation: existing
  docker prune cron / disk-monitoring on the agent should
  absorb this; if not, add a periodic `docker image prune
  --filter='until=168h'` housekeeping cron.
- **Tag drift between parent and downstream.** The
  parent's pushed tag is its BUILD_NUMBER; the downstream
  receives that as UPSTREAM_BUILD. As long as the parent
  pipeline doesn't change tag scheme without coordinating
  here, this is invariant. The parent's
  `:${BUILD_NUMBER}` tag is always the freshest.
